# Git Commit

分析代码变更，以合适的粒度创建符合规范（Conventional Commit）且带有 emoji 的提交。

> ⚠️ **所有 commit message（包括描述部分）必须使用中文书写。**

## 工作流程

### 第一步：检查变更状态

运行 `git status`。如果没有任何变更，退出。

根据文件状态决定分析范围：

| 状态 | 操作 |
|------|------|
| 同时存在已暂存 + 未暂存 | 询问用户是否一起提交；同意则 `git add .`，否则只分析已暂存部分 |
| 只有未暂存变更 | 执行 `git add .`，分析所有变更 |
| 只有已暂存变更 | 直接分析已暂存变更 |

### 第二步：分析变更内容

运行 `git diff --cached` 获取将要提交的完整 diff，理解变更意图。

### 第三步：决定是否拆分

参考下方"拆分准则"，自主判断是否需要多个 commit：
- 需要拆分 → 制定分组计划，逐组暂存并提交
- 无需拆分 → 直接单次提交

### 第四步：执行提交

按照"commit 格式规范"构造 message，执行 `git commit -m "<message>"`。

## Commit 格式规范

格式：`<emoji> <type>: <中文描述>`

- 描述必须使用**中文**
- 第一行不超过 **100 个字符**
- type 与 emoji 对照见下方"常用 Emoji 对照表"

### 常用 type

| type | 含义 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `docs` | 文档更改 |
| `style` | 代码格式/样式（不影响逻辑） |
| `refactor` | 重构（非 fix 非 feat） |
| `perf` | 性能优化 |
| `test` | 测试相关 |
| `chore` | 构建、工具、依赖等杂项 |
| `ci` | CI/CD 相关 |

## 拆分准则

满足以下任一条件时，应拆分为多个 commit：

1. **不同关注点**：变更涉及代码库中相互独立的模块
2. **不同变更类型**：同时包含功能开发、Bug 修复、重构等
3. **文件类型差异**：源代码与文档、配置混在一起
4. **逻辑上可独立审查**：分开后更易理解和 review
5. **体量过大**：拆分后语义更清晰

## 示例

单次提交：
```
✨ feat: 添加用户登录功能
🐛 fix: 修复金额精度计算错误
♻️ refactor: 简化错误处理逻辑
📝 docs: 更新 API 接口文档
```

拆分提交：
```
第 1 个：✨ feat: 添加新的合约版本类型定义
第 2 个：📝 docs: 更新对应版本的文档说明
第 3 个：🔧 chore: 升级相关依赖版本
第 4 个：✅ test: 为新版本功能添加单元测试
```
