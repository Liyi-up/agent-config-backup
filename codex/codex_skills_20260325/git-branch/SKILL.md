# Git Branch

分析本地代码变更，推断最能描述变更意图的分支名，并直接创建该分支。

## 工作流程

### 第一步：检查变更状态

运行 `git status`。如果没有任何变更（已暂存 + 未暂存均为空），提示用户无法推断分支名，退出。

### 第二步：获取变更内容

```bash
git diff HEAD        # 未暂存变更
git diff --cached    # 已暂存变更
```

综合两部分 diff 与 `git status` 中的文件名，推断本次变更的主题和意图。

### 第三步：生成分支名

按照下方"命名规范"生成分支名。若变更涉及多个不相关主题，取**最主要**的那个，并在创建后告知用户。

创建前检查是否存在同名分支；若冲突，在描述末尾追加数字后缀（如 `-2`）。

### 第四步：创建分支

```bash
git checkout -b <branch-name>
```

创建成功后输出最终分支名，供用户确认。

## 命名规范

格式：`<type>/<short-description>`

### type 选项

| type | 含义 |
|------|------|
| `feat` | 新功能开发 |
| `fix` | Bug 修复 |
| `docs` | 文档更新 |
| `style` | 代码样式/格式调整 |
| `refactor` | 代码重构 |
| `perf` | 性能优化 |
| `test` | 测试相关 |
| `chore` | 构建、工具、依赖等杂项 |
| `ci` | CI/CD 相关 |
| `experiment` | 实验性探索 |

### short-description 规则

- 使用**英文小写**，单词间用 `-` 连接（kebab-case）
- **优先简短**：通常控制在 **2~3 个单词**，能在不丢失语义的前提下更短就更短
- 严格控制长度：建议不超过 **20 个字符**
- 准确体现核心意图，避免模糊词（`update`、`change`、`misc`）
- 禁止空格、下划线、大写字母或特殊字符

### 示例

| 变更意图 | 推荐分支名 |
|----------|-----------|
| 新增用户登录功能 | `feat/user-login` |
| 修复支付金额计算错误 | `fix/payment-amount` |
| 重构数据库查询逻辑 | `refactor/db-query-logic` |
| 更新 README 文档 | `docs/readme` |
| 优化首页加载性能 | `perf/home-load` |
| 升级依赖版本 | `chore/bump-deps` |
| 实验新的缓存方案 | `experiment/cache-plan` |

---

## 注意事项

- 此操作**只创建分支**，不影响暂存区或提交历史
- 变更内容仅用于推断意图，不会被提交
